Electron 使用心得

发表于 2021-01-31 更新于 2021-02-01 分类于 2701：Electron

Electron#

一、项目构建和打包#

项目搭建：
electron-vue

项目打包：
electron-builder

  • 可以支持 Windows/Mac/Linux 平台
  • 支持自动更新，打包更轻量
  • 可以打包出不暴露源码的安装程序
    electron-packager
  • 配置 nsis 自定义安装目录以及安装个性化

二、Electron 知识点#

1、主进程和渲染进程#

主进程：package.json 中的 main.js
Electron 内核是 Chromium
在主进程中通过 new BrowserWindow 创建页面，每个 BrowserWindow 就是渲染进程，当某个页面被销毁后，对应的渲染进程就会终止。

2、优化窗口#

（1）无边框窗口

mainWindow = new BrowserWindow({width: 800, height: 600, frame: false, resizable: false});

（2）窗口拖动
现在窗口已经没有标题栏和菜单栏了，所以窗口没办法拖动了，给拖动区域增加 -webkit-app-region: drag 的 CSS 代码。
<body style="-webkit-app-region: drag"></body>
需要注意的是，如果在拖动区域里有按钮等需要点击的元素，需要增加 -webkit-app-region: no-drag 设置，否则会无法点击。

3、进程间通信#

因为安全限制，页面不能直接访问原生 GUI 资源，在 Electron 中也是一样。
渲染进程要进行原生的 GUI 操作，比如和主进程进行通信，请求相应的操作，在 Electron 中提供了几种通信方式：

（1）ipcMain 和 ipcRenderer
在渲染进程中使用 ipcRenderer 模块来向主进程发送消息，主进程中使用 ipcMain 模块接受消息，进行操作。如果还需要进行反馈，则通知渲染进程，渲染进程根据接收的内容执行对应的操作。

// 渲染进程中
const { ipcRenderer } = require("electron");
ipcRender.send("somemsg", data);
ipcRender.on("replaymsg", (evt, otherData) => {
  console.log(otherData);
});
// 主进程中
const { ipcMain } = require("electron");
ipcMain.on("somemsg", (evt, data) => {
  console.log(data);
  evt.sender.send("replymsg", otherData);
});

（2）remote
remote 模块为渲染进程和主进程通信提供了一种简单方法

（3）webContents

// 主进程中
win.webContents.on("did-finish-load", () => {
  win.webContents.send("ping", "23333333333");
});
// 渲染进程中
require("electron").ipcRenderer.on("ping", (event, message) => {
  console.log(message);
});

（4）渲染进程之间通信
使用 Vuex 和 router
但如果对数据实时性要求高，那就要结合上面的方法来实现：

// 主进程中，将两个窗口的 id 分别发送到对方的渲染进程里
mainWindow.webContents.on("did-finish-load", () => {
  mainWindow.webContents.send("getChild", {
    childId: childWindow.id,
  });
});
childWindow.webContents.on("did-finish-load", () => {
  childWindow.webContents.send("getMain", {
    mainId: mainWindow.id,
  });
});

// 渲染进程 mainWindow，通过 id 得到 childWindow 并发送一条消息
ipcRenderer.on("getChild", (event, arg) => {
  remote.BrowserWindow.fromId(arg.childId).webContents.send(
    "say",
    "Hello World!"
  );
});
// 渲染进程 childWindow，接收 mainWindow 的消息
ipcRenderer.on("say", (event, arg) => {
  console.log(arg); // 输出：Hello World!
});
// 同理 childWindow 也可以通过 id 给 mainWindow 发送消息

4、踩坑#

electron-vue 里使用 elementUI 等，需要在 webpack.renderer.config.js 的白名单里配置：let whiteListedModules = ['vue', 'element-ui']

5、难点#

多窗口的管理，每个窗口使用 remote.getCurrentWindow().id 命名，在主进程中采用单个实例 app.makeSingleInstance 管理

03、创建主脚本文件 main.js

const { app, BrowserWindow } = require("electron");

// 创建一个新的窗口
function createWindow() {
  const win = new BrowserWindow({
    width: 800,
    height: 600,
    webPreferences: {
      nodeIntegration: true,
    },
  });
  win.loadFile("index.html");
}

app.whenReady().then(createWindow);

// 当程序不再有任何打开窗口时试图退出
app.on("windows-all-closed", () => {
  if (process.platform !== "darwin") {
    app.quit();
  }
});

// 只有当应用程序激活后没有可见窗口才能创建新的窗口
app.on('activate', ()=>{
  if(BrowserWindow.getAllWindows().length === 0){
    createWindow()
  }
})

Vuex#

管理多组件共享的状态数据。
核心概念：

  • store
  • state
    用于存储多组件共享状态
  • mutation
    函数
    直接修改 state 的数据
    必须写同步代码（为什么？ 官方代码规范 / Vue devTool 记录触发过的 mutation，如果写入异步代码，会导致 mutation 的执行顺序和更新顺序不统一，调试工具就报废了）
  • action
    函数
    通过 mutation 来间接修改状态数据。
    可以是异步函数
  • getter
    函数（用的是它的返回值）
    相当于计算属性（当需要一个数据，但是没有，然而这个数据和 data 或者 props 相关，最终可以通过他们获得）
  • dispatch
    用于分发 action
  • commit

发表于 2021-01-30 更新于 2021-02-01 分类于 2404：前端性能优化

防抖与节流#

一、节流#

防抖就是防止把一次事件触发多次，一般用在登录、发短信、调整窗口大小，文本编辑器的实时保存。
假设有一个场景，滚动就会发起网络请求，但是我们不希望用户在滚动过程中一直发请求，而是隔一段时间发一次，对于这种情况，我们可以使用节流。

二、防抖#

节流就是控制事件的发生频率，一般用在滚动事件，播放器进度条计算，实时搜索请求。
假设有一个按钮，用户点击一次就会触发网络请求，但是我们不希望用户一直点这个按钮发请求，而是这个用户点击这个按钮一段时间内没有再去点击才去发请求，对于这种情况，我们可以使用防抖。

Webpack#

自动化构建工具：开发环境 --> 生产环境

开发环境（development）注重开发效率和源码的阅读。

生产环境（production）注重代码的执行效率（压缩）。

gulp / grunt 属于任务流工具（Task Runner）

webpack 是模块化打包工具（bundler）

webpack 是一个模块打包器（bundler），根据模块的依赖关系进行静态分析，生成对应的静态资源。对 webpack 来说，前端所有资源文件（js/json/css/less/img/…）都会作为模块处理。

核心概念

Entry 入口起点，指示 webpack 应该使用哪个模块，来作为构建其内部依赖图的开始。

Output 告诉 webpack 在哪里输出它所创建的 bundles，以及如何命名这些文件。

Loader 让 webpack 能够处理那些非 JS 文件。

Plugins 插件，用于执行范围更广的任务，比如打包优化，压缩。

Mode 模式，有生产模式（production）和开发模式（development）。

1、开始#

（1）初始化

npm init

（2）安装

# 全局安装
npm i webpack webpack-cli -g

# 局部安装（建议）

demo
 ├── src
 │   ├── js

 ├── build
 ├── package.json

（1）打包 JS 文件

能够编译打包 js 和 json 文件，并且能将 es6 的模块化语法转换成浏览器能识别的语法。

将 src/js/app.js 打包成开发环境文件 build/js/app.js

npx webpack src/js/app.js -o build/js/app.js --mode=develoment
#                         -output

打包到生产环境（比生产环境多了一个代码压缩）

npx webpack src/js/app.js -o build/js/app.js --mode=production

缺点就是不能编译打包 CSS img 等文件，不能将 ES6 基本语法转化成 ES5 以下语法。

可以通过配置文件改善。

# 对指定文件进行打包
npx webpack xxx.js --output xxx.js

# 打包多个文件
npx webpack x1.js x2.js

（2）默认配置文件 webpack.config.js △

const { resolve } = require("path");

module.exports = {
  // 指定打包文件的入口
  // （单入口写法）
  // entry: ".src/js/app.js",
  // （多入口写法）
  entry: {
    app: "./src/js/app.js",
    mod: "./src/js/mod.js",
  },

  // output
  output: {
    // 指定打包之后的名字
    //               ↓ entry 的属性名
    filename: "js/[name].index.js",
    // 指定打包的绝对路径
    path: resolve(__dirname, "./build"),
  },

  // 指定打包模式
  // - development 开发模式
  // - production  生产模式
  mode: "development",
};

跨域解决方案

发表于 2021-01-30 更新于 2021-02-01 分类于 2505：网络请求

跨域解决方案#

一、同源策略#

Same-Origin Policy，同源策略 最早由 Netscape 公司提出，是浏览器的一种安全策略。该策略表示协议、域名、端口号必须完全相同。

违背同源策略就是跨域，主要是针对 ajax 而言。

二、JSONP#

JSON with Padding，JSONP 是一个非官方的跨域解决方案，只支持 get 请求，而且需要服务器的配合。

1、工作原理#

在网页有一些标签天生具有跨域能力，比如 img link iframe script。JSONP 就是利用 script 标签的跨域能力来发送请求的。

2、应用示例#

客户端

<script>
  const script = null;
  // 这是一个事件触发
  ..{
      // 新建一个 script 标签
      script = document.createElement("script");
      // 设置需要跨域的URL
      script.src = "http://.../test?a=1&cb=jp";
      // 插入到页面
      document.body.appendChild(script);
  }
  // 回调函数
  function jp(n){
      // 这边要移除script 标签
      document.body.removeChild(script);
      // 后续操作
  }
</script>

服务器

res.end(cb(xxx));

三、CORS#

Cross-Origin Resource Sharing，CORS，跨域资源共享是官方的跨域解决方案，它的特点是不需要在客户端做任何特殊的操作，完全在服务器中进行处理，支持 GET 和 POST 请求。是通过设置一个响应头来告诉浏览器，该请求允许跨域，浏览器收到该响应以后就会对响应放行。

// 服务端
app.all("*", (req,res)=>{
    // 设置白名单
    const arrOrigin = [
        "http://xxxxx",
        "xxxx"
    ]
    // 如果访问的URL在白名单中
    if(arrOrigin.oncludes(req.header.origin)){
        // 允许跨域
        res.header("Access-Control-Allow-Origin"),req.header.origin);
        // 允许携带内容
        res.header("Access-Control-Allow-Headers", "content-type")
    }
    next();
})

四、服务器代理#

服务器和服务器之间没有同源策略限制，所以可以通过代理的方式进行跨域。

兼容性不好。

      atk: 999,
      parry: 999
    },
  }),

  .then((response) => response.json())
  .then((res) => {
